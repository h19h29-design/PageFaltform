# Dispatch Wave 03

기준일: `2026-03-21`

이 wave의 목표는 `로컬에서 거의 완성품처럼 테스트 가능한 수준`까지 끌어올리는 것입니다.

이번 wave가 끝나면 최소한 아래가 가능해야 합니다.

- 가입 또는 운영자 계정 생성
- 로그인 / 로그아웃
- 가족 홈 진입
- 모든 모듈 페이지 이동
- 게시판/일정/트래커 항목 생성, 수정, 삭제
- 콘솔에서 가족 홈과 모듈 구성을 바꾸고 다시 확인

## HQ 판단

- 기존 wave는 `계약`, `builder`, `합성`, `DB 준비` 단계였습니다.
- 이번 wave는 `실제 페이지`, `CRUD`, `가입/세션`, `테스트` 단계입니다.
- 메인 기반 작업자는 `04`와 `05`입니다.
- 실질적인 제품 화면 채우기는 `02`, `03`, `06`이 병렬로 맡습니다.
- `01`은 계약 감리와 최종 QA 정리를 맡습니다.

## 저장/인증 기본 가정

- 이번 wave는 `로컬 PostgreSQL + Prisma`를 기본 경로로 가정합니다.
- DB가 준비되지 않은 환경에서는 최소한의 demo/file fallback이 남을 수 있지만, 목표는 `DB source of truth`입니다.
- 이번 wave가 끝나면 사용자는 로컬에서 가입, 생성, 수정, 삭제를 직접 시험할 수 있어야 합니다.

## Thread 04

이번 wave 목표:

- 실제 가입/로그인/세션과 DB source of truth를 연결합니다.

이번 스레드에서 해야 할 일:

1. 로컬에서 쓸 `sign-up`, `sign-in`, `sign-out` 흐름을 만듭니다.
2. demo/file 인증 흐름을 DB 사용자와 membership을 읽는 구조로 바꿉니다.
3. family access session도 DB session row를 쓰는 쪽으로 옮기기 시작합니다.
4. bootstrap/import 명령과 웹 런타임 사이를 실제로 연결합니다.
5. 로컬 테스트용 기본 운영자 / 일반 사용자 / 가족 membership 시나리오가 가능하게 만듭니다.

완료 기준:

- 로컬에서 계정 생성과 로그인 테스트가 됩니다.
- 콘솔이 DB 사용자 기준으로 동작합니다.
- 가족 입장 세션도 DB 기반 전환 시작점이 코드에 반영됩니다.

## Thread 05

이번 wave 목표:

- 가족 앱 전체 쉘과 페이지 라우팅을 실제 제품 구조처럼 보이게 만듭니다.

이번 스레드에서 해야 할 일:

1. `app/[familySlug]` 내부에 전체 모듈 메뉴와 이동 구조를 정리합니다.
2. 모든 모듈 페이지로 이동 가능한 route map을 만듭니다.
3. 콘솔과 가족 홈 사이를 오가는 운영 흐름을 정리합니다.
4. list / detail / new / edit 페이지 쉘 패턴을 공통으로 맞춥니다.
5. 실제 로컬 테스트 동선에서 길이 끊기지 않도록 빈 페이지를 없앱니다.

완료 기준:

- 사용자가 가족 홈에서 모든 모듈 페이지로 이동할 수 있습니다.
- 콘솔, 입장, 홈, 상세 페이지 동선이 끊기지 않습니다.
- 페이지가 단순 404나 placeholder가 아니라 실제 제품처럼 보입니다.
