# 가족 플랫폼 제품 방향

## 1. 제품 정의

이 프로젝트는 "우리 가족 홈페이지"에서 시작하지만, 최종적으로는 누구나 자신의 가족 공간을 만들어 쓸 수 있는 플랫폼을 목표로 합니다.

핵심은 다음 3가지입니다.

- 가족마다 독립된 공간을 가진다.
- 각 공간은 필요한 모듈만 켜서 사용한다.
- 각 모듈에서 고른 핵심 정보가 메인 홈으로 올라온다.

## 2. 주요 사용자

- 가족 관리자: 가족 공간 생성, 테마 변경, 메뉴 구성, 멤버 초대, 권한 설정
- 일반 가족 구성원: 글쓰기, 사진 업로드, 일정 관리, 투두 체크
- 제한 사용자: 읽기 전용 또는 일부 게시판만 접근
- 플랫폼 운영자: 전체 테넌트 관리, 플랜 관리, 도메인 관리, 장애 대응

## 3. 핵심 사용자 흐름

### 첫 진입

- 가족별 스플래시 화면 진입
- `들어가기` 버튼 클릭
- 비밀번호 또는 접근 코드 입력
- 성공 시 가족 홈으로 이동

### 가족 홈

- 오늘의 일정
- 오늘의 할 일
- 오늘의 진행률 또는 습관 체크
- 최근 공지
- 최근 사진
- 다가오는 기념일
- 메인 노출로 체크된 게시물/카드

### 모듈 사용

- 각 게시판은 자기 화면에서 글/항목을 생성
- 작성 시 `메인에 보이기` 또는 `고정` 체크 가능
- 체크된 항목은 메인 대시보드 집계 규칙에 따라 카드로 노출

## 4. 모듈 체계

### 공통 게시판/모듈 타입

- 콘텐츠형: 공지, 일반 글, 일기, 가족 규칙
- 미디어형: 갤러리, 추억 보관함, 파일 보관함
- 체크형: 투두, 장보기 리스트, 습관 체크
- 시간형: 일정, 학교/학원 시간표, 24시간 하루 시간표
- 트래커형: 진행률 보드, 목표 관리, 저축/공부/운동 진행 현황

### 1차 필수 MVP

- 공지 게시판
- 일반 글 게시판
- 갤러리
- 투두 리스트
- 일정/캘린더
- 메인 대시보드
- 가족 진입 화면과 비밀번호 접근

### 1.5차 생활형 MVP

- 일기
- 학교/학원 시간표
- 24시간 하루 시간표
- 진행률 보드
- 습관 체크

### 2차 확장 모듈

- 장보기 리스트
- 식단표
- 기념일/이벤트
- 추억 보관함
- 가족 규칙/가이드
- 설문/투표
- 파일 보관함
- 가계부 라이트

## 5. 제품 원칙

### 플랫폼 원칙

- 가족 하나를 위한 맞춤형 사이트처럼 보여야 한다.
- 동시에 여러 가족이 재사용 가능한 공통 구조여야 한다.
- 모듈을 켜고 끄는 설정이 쉬워야 한다.
- 같은 코드베이스로 여러 가족을 운영할 수 있어야 한다.

### UX 원칙

- 첫 화면은 감성적이고 따뜻해야 한다.
- 메인 홈은 복잡하지 않고 "오늘 바로 봐야 할 것" 중심이어야 한다.
- 글쓰기 흐름은 단순해야 한다.
- 모바일 사용성이 우선이어야 한다.

### 운영 원칙

- 관리자용 보안과 가족 공유용 접근 코드는 분리한다.
- 테넌트 데이터는 확실하게 분리한다.
- 테마/메뉴/모듈 설정은 데이터 기반으로 관리한다.
- 지금 다 만들지 않아도 되도록, 새 모듈은 공통 계약 위에 추가 가능해야 한다.

## 6. 메인 홈 구성 제안

- 히어로 카드: 가족 인사말, 오늘 날짜, 날씨 또는 대표 메시지
- 오늘 카드: 오늘 일정, 오늘 투두, 오늘 시간표
- 진행 카드: 진행률, 습관 연속 기록, 목표 달성률
- 최신 카드: 최근 공지, 최근 게시글, 최근 사진
- 특별 카드: 생일, 기념일, D-day
- 고정 카드: 관리자가 메인 노출 체크한 게시물

## 7. 공지 게시판 기본 양식

- 제목
- 공지 유형: 일반 / 중요 / 긴급
- 상단 고정 여부
- 메인 노출 여부
- 메인 노출 우선순위
- 노출 시작일 / 종료일
- 대상 범위: 전체 / 특정 구성원 그룹
- 본문
- 첨부파일
- 댓글 허용 여부
- 읽음 확인 여부

## 8. 진행률/트래커 모듈 아이디어

- 퍼센트형: 공부 진도 70%, 저축 목표 40%
- 개수형: 책 3/10권, 청소 5/7회
- 연속형: 운동 12일 연속, 일기 8일 연속
- 단계형: 여행 준비 1단계 완료, 2단계 진행 중

메인 홈에서는 다음처럼 활용합니다.

- 이번 주 목표 달성률
- 오늘의 루틴 완료 수
- 진행 중인 가족 프로젝트 현황

## 9. 비기능 요구사항

- 모바일 우선 반응형
- 이미지 업로드 최적화
- 게시물과 일정 검색 가능
- 테넌트별 테마 적용
- 향후 커스텀 도메인 연결 가능
- 배포 자동화 가능

## 10. 지금 다 만들지 않아도 되는 이유

- 게시판과 모듈을 공통 타입으로 나누면 나중에 추가 비용이 줄어든다.
- 메인 홈 노출 규칙을 먼저 통일하면 어떤 모듈이든 같은 방식으로 붙일 수 있다.
- 공지, 일정, 투두, 일반 글만 먼저 만들어도 홈 화면의 플랫폼 감은 충분히 살아난다.

## 11. 지금 당장 하지 않을 것

- 처음부터 모바일 앱 동시 개발
- 처음부터 복잡한 결제 플랜
- 처음부터 마이크로서비스 분해
- 초기에 너무 많은 자동화/알림 채널
