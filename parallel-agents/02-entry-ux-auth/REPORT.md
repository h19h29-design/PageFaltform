- 가족 진입은 감성적인 첫 화면 -> 짧은 접근 확인 -> 가족 홈 진입의 3단 구조가 가장 자연스럽다.
- 관리자 기능은 가족 홈 안에 숨어 있는 버튼이 아니라, 별도 계정 인증과 별도 보호 경로로 분리하는 편이 안전하다.

## 1. 임시 기본값

사용자 확정 전까지 아래 가정으로 진행하는 것이 재작업을 줄인다.

- 첫 화면 톤: 따뜻하고 차분한 가족 앨범 느낌, 크림/베이지 바탕 + 짙은 올리브/차콜 포인트
- 첫 화면 구성: 가족명, 한 줄 환영 문구, 커버 이미지 또는 일러스트, `들어가기` 1차 CTA
- 가족 입장 방식: 기본은 가족 공용 비밀번호 1개, 필요 시 초대용 접근 코드 확장 가능
- 로그인 없이 허용할 범위: 스플래시 진입, 접근 코드/비밀번호 입력, 가족 홈 읽기 중심 사용
- 관리자 보호: 무조건 계정 로그인 기준, 가족 공용 비밀번호만으로는 진입 불가

## 2. UX 방향

### 첫 화면 시안 방향

첫 화면은 "서비스 소개"보다 "우리 가족 공간에 들어오는 입구"처럼 보여야 한다. 너무 기능적이면 감성이 사라지고, 너무 장식적이면 다음 행동이 흐려진다. 따라서 아래 3요소를 우선 고정하는 편이 좋다.

1. 한 번에 이해되는 정체성
2. 누르면 바로 들어갈 수 있을 것 같은 단일 CTA
3. 실패해도 당황하지 않는 짧은 인증 흐름

추천 구성:

- 상단: 가족명 또는 서비스명
- 중단: 커버 비주얼, 짧은 환영 문구, 오늘 날짜나 계절감 있는 보조 문장
- 하단: `들어가기` 버튼, 보조 텍스트 버튼 `관리자 로그인`

모바일 기준 화면 메모:

- 첫 진입 화면은 한 손 조작 기준으로 CTA를 하단에 둔다.
- 접근 비밀번호 입력은 전체 페이지 전환보다 바텀시트 또는 카드 전환이 더 자연스럽다.
- 오류 메시지는 팝업보다 입력창 바로 아래 인라인 메시지가 낫다.

## 3. 진입 플로우 정의

### 라우트 책임 초안

| 경로 | 목적 | 인증 상태 |
| --- | --- | --- |
| `/` | 플랫폼 기본 랜딩 또는 가족 slug 진입 안내 | 비로그인 |
| `/f/[familySlug]` | 가족 전용 스플래시 및 접근 확인 | 비로그인 가능 |
| `/app/[familySlug]` | 가족 홈 및 가족 모듈 | 가족 접근 세션 필요 |
| `/console/sign-in` | 관리자 계정 로그인 | 비로그인 |
| `/console` | 관리자 설정/운영 화면 | 관리자 계정 세션 필요 |

추가 메모:

- 커스텀 도메인 도입 시 도메인 루트가 곧 `/f/[familySlug]` 역할을 하도록 매핑하면 된다.
- 관리자 기능은 당분간 `/app/[familySlug]` 안에 섞지 말고 `/console` 계열로 분리하는 편이 낫다.

### 플로우 A. 가족 진입

1. 사용자가 `/f/[familySlug]`에 진입한다.
2. 가족 스플래시를 본다.
3. `들어가기`를 누른다.
4. 가족 접근 비밀번호 또는 접근 코드를 입력한다.
5. 검증 성공 시 `family access session`을 만든다.
6. `/app/[familySlug]`로 이동한다.

핵심 UX 포인트:

- 사용자는 "로그인했다"보다 "입장 확인을 마쳤다"는 느낌을 받아야 한다.
- 성공 직후에는 로더보다 짧은 환영 전환 문구가 있는 편이 감성에 맞다.

### 플로우 B. 재방문

1. 기존 `family access session`이 유효하면 입력 단계를 건너뛴다.
2. 바로 `/app/[familySlug]` 또는 직전 위치로 보낸다.
3. 세션이 만료되었으면 접근 확인 화면으로 되돌린다.

### 플로우 C. 관리자 진입

1. 사용자가 스플래시 하단 또는 별도 경로에서 `관리자 로그인`을 선택한다.
2. `/console/sign-in`으로 이동한다.
3. 계정 인증을 통과하면 관리자 세션을 만든다.
4. 가족 선택 또는 최근 작업 가족으로 `/console`에 진입한다.

핵심 분리 원칙:

- 가족 공용 비밀번호는 "가족 공간 접근"만 증명한다.
- 관리자 계정 로그인은 "누가 관리 작업을 하는지"를 증명한다.
- 같은 화면에서 둘을 동시에 요구하지 않는다.

### 플로우 D. 실패/예외

상태는 MVP부터 명시해 두는 편이 좋다.

- 잘못된 비밀번호 또는 코드
- 만료된 접근 코드
- 너무 많은 시도 후 일시 제한
- 비활성 가족 또는 존재하지 않는 slug
- 세션 만료 후 재인증 필요
- 관리자 계정은 유효하지만 해당 가족 관리 권한이 없음

## 4. 세션과 인증 분리 초안

### 세션 종류

| 세션 | 용도 | 저장 정보 | 비고 |
| --- | --- | --- | --- |
| `family access session` | 가족 공간 읽기/기본 사용 | `familySlug`, `tenantId`, 접근 수준, 만료 시각 | 공용 비밀번호/접근 코드 기반 |
| `platform user session` | 계정 식별, 역할 판별, 관리자 보호 | `userId`, 소속 가족, 역할 | 관리자 및 멤버 계정 로그인 기반 |

기본 원칙:

- `family access session`만으로는 관리자 화면 접근 불가
- `platform user session`이 있더라도 가족 접근 정책이 막혀 있으면 해당 가족 홈 진입은 별도 판단
- 향후에는 두 세션을 조합해 `guest -> member` 업그레이드 UX를 만들 수 있다

### 추천 검증 함수 방향

`packages/auth` 기준으로 아래 책임이 필요하다.

- `verifyFamilyAccess(...)`
- `createFamilyAccessSession(...)`
- `getFamilyAccessSession(...)`
- `requirePlatformUser(...)`
- `requireFamilyRole(...)`
- `canAccessConsole(...)`

`packages/tenant` 기준으로는 아래 책임이 자연스럽다.

- `resolveFamilyFromSlug(...)`
- `resolveFamilyFromDomain(...)`
- `getActiveFamilyContext(...)`

## 5. 권한 UX 초안

### 역할 모델 기본안

| 역할 | 획득 방식 | 기본 의도 |
| --- | --- | --- |
| `owner` | 계정 로그인 + 가족 소유자 | 가족 설정과 권한 전체 관리 |
| `admin` | 계정 로그인 + 관리자 지정 | 콘텐츠/일정/운영 관리 |
| `member` | 계정 로그인 + 가족 멤버 | 일반 작성과 참여 |
| `guest` | 가족 공용 비밀번호/접근 코드 | 읽기 중심 진입 |
| `child` | 계정 로그인 또는 간편 진입 확장안 | 제한된 모듈만 사용 |

### MVP 권한 기본안

| 행동 | guest | child | member | admin | owner |
| --- | --- | --- | --- | --- | --- |
| 가족 홈 보기 | 가능 | 가능 | 가능 | 가능 | 가능 |
| 공지 읽기 | 가능 | 가능 | 가능 | 가능 | 가능 |
| 일반 글/사진 보기 | 가능 | 가능 | 가능 | 가능 | 가능 |
| 글/사진 작성 | 불가 | 제한적 | 가능 | 가능 | 가능 |
| 투두/습관 체크 | 정책 기반 | 가능 | 가능 | 가능 | 가능 |
| 모듈 설정 변경 | 불가 | 불가 | 불가 | 일부 | 가능 |
| 가족 멤버/권한 관리 | 불가 | 불가 | 불가 | 불가 | 가능 |
| 관리자 콘솔 접근 | 불가 | 불가 | 불가 | 가능 | 가능 |

메모:

- `guest`는 로그인 없이 허용되는 범위를 담는 역할로 본다.
- `child`는 MVP에서 지나치게 복잡하게 만들지 말고, 우선 `member`보다 좁은 읽기/체크 중심 역할로 정의하는 편이 낫다.
- `admin`은 가족 홈의 일반 사용자가 아니라 운영 권한 보유자이므로, 콘솔 분리가 UX와 보안 모두에 유리하다.

## 6. 모바일 우선 화면 상태

### S1. 가족 스플래시 `/f/[familySlug]`

- 기본: 비주얼, 환영 문구, `들어가기`
- 로딩: 가족 테마/커버 로딩 스켈레톤
- 오류: 존재하지 않는 가족, 비활성 가족

### S2. 접근 확인

- 기본: 비밀번호 또는 코드 입력
- 오류: 불일치, 만료, 잠시 후 재시도
- 보조 액션: `관리자 로그인`

### S3. 가족 홈 진입 직전

- 성공 피드백: 짧은 전환 문구 또는 페이드
- 실패: 세션 생성 실패, 네트워크 오류

### S4. 관리자 로그인 `/console/sign-in`

- 기본: 이메일/로그인 식별자 + 비밀번호
- 오류: 인증 실패
- 후속: 권한 없는 계정 안내

### S5. 보호된 화면

- 가족 홈 보호: 가족 접근 세션 없으면 `/f/[familySlug]`로 회귀
- 콘솔 보호: 관리자 계정 세션 없으면 `/console/sign-in`으로 회귀

## 7. 필요한 화면 목록

MVP 기준 필수 화면은 아래 정도로 잡는 것이 적당하다.

1. 플랫폼 랜딩 또는 slug 안내 화면
2. 가족 스플래시 화면
3. 가족 접근 비밀번호/코드 입력 화면
4. 가족 홈 화면
5. 관리자 로그인 화면
6. 관리자 콘솔 진입 화면
7. 접근 실패/가족 없음 화면
8. 세션 만료 재진입 화면

선택 화면:

1. 가족 선택 화면
2. `guest -> member` 전환 안내 화면
3. 접근 코드 재발급 또는 만료 안내 화면

## 8. 패키지별 구현 포인트

### `apps/web`

- 라우트 그룹을 `public`, `family`, `console` 관점으로 나누는 편이 이후 유지보수에 유리하다.
- 최소 라우트는 `/`, `/f/[familySlug]`, `/app/[familySlug]`, `/console/sign-in`, `/console`를 먼저 만든다.

### `packages/auth`

- 가족 접근 검증과 계정 인증을 같은 함수/미들웨어에 섞지 않는다.
- 역할 판별은 `tenant` 컨텍스트를 받은 뒤 수행하는 구조가 안전하다.

### `packages/tenant`

- slug 기반 해석을 먼저 고정하고, 커스텀 도메인은 같은 인터페이스로 확장한다.
- 현재 활성 가족 컨텍스트를 반환하는 공용 유틸이 필요하다.

### `packages/ui`

- 첫 화면과 인증 카드에 공용 비주얼 톤을 넣을 수 있도록 컬러 토큰과 카드/버튼 베이스를 먼저 잡는다.
- 스플래시, 접근 카드, 오류 상태를 서로 다른 제품처럼 보이지 않게 묶어야 한다.

## 9. 사용자 확인이 필요한 항목

아래는 바로 구현을 막을 수준은 아니지만, 확정되면 재작업을 더 줄일 수 있다.

1. 첫 화면 분위기를 사진 중심으로 갈지, 일러스트 중심으로 갈지
2. 가족 공용 비밀번호를 고정 비밀번호로 둘지, 초대 코드 중심으로 갈지
3. 로그인 없이 허용할 범위를 읽기 전용으로 제한할지, 체크/댓글까지 열지
4. `child` 역할을 MVP부터 분리할지, 1차에서는 `member` 축소판으로 둘지
