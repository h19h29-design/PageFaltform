# Role Permissions Draft

목적:

- `02 Entry/Auth`와 `06 Platform/Admin`이 같은 기준으로 권한을 해석하도록 초안 기준선을 만든다.
- 이 문서는 MVP v1 기준 권한 초안이며, 사용자 확인 전까지의 임시 합의안이다.

## 1. 역할 정의

### 가족 테넌트 역할

- `owner`: 가족 공간 최상위 관리자
- `admin`: 운영 보조 관리자
- `member`: 일반 가족 구성원
- `guest`: 읽기 중심 사용자
- `child`: 제한 모듈 사용자

### 플랫폼 역할

- `platformOperator`: 플랫폼 전체 운영자

메모:

- `platformOperator`는 가족 내부 역할과 별개다.
- MVP 기준 핵심 구현 대상은 가족 테넌트 역할 5종이다.

## 2. 권한 해석 원칙

1. 가족 진입 비밀번호와 관리자 권한은 분리한다.
2. 관리자 보호 기능은 계정 인증 기준으로 판단한다.
3. `guest`, `child`는 읽기 우선이며, 예외 허용만 추가한다.
4. 메인 노출 설정은 일반 작성 권한과 분리한다.
5. 삭제 권한은 생성 권한보다 더 엄격하게 준다.

## 4. 모듈별 해석 메모

### 공지 게시판

- 공지는 전달 책임이 있으므로 `owner`, `admin`만 작성한다.
- 읽음 확인이 켜진 공지는 읽기 가능한 사용자가 확인 처리할 수 있다.
- 댓글은 기본 비활성이며, 켜더라도 `guest`는 제외하는 안이 안전하다.

### 일반 글/일기/갤러리

- `member`는 자신의 콘텐츠를 생성하고 관리할 수 있다.
- `child`는 허용된 보드 또는 앨범에만 업로드 가능하도록 연다.
- 일기는 공개 범위 정책이 확정되기 전까지 `member` 이상만 작성 가능으로 시작한다.

### 일정/투두

- MVP에서는 가족 공용 관점이 강하므로 `member`까지 작성 허용으로 둔다.
- 담당자 기능이 들어가면 `Own` 규칙을 세분화할 수 있다.

### 진행률/습관

- 체크 행위는 넓게 열되, 정의와 삭제는 좁게 두는 편이 안전하다.
- 초기에는 `member`가 체크 가능, `owner/admin`이 템플릿 관리 가능 구조가 적합하다.

## 5. `child` 역할 기본안

- 접근 가능한 모듈은 가족 설정에서 화이트리스트로 고른다.
- 수정 가능한 행동은 체크형 또는 자기 기록형으로 제한한다.
- 공지와 일정은 읽기 중심으로 둔다.
- 가족 설정, 멤버 관리, 메인 노출 설정은 항상 금지한다.

## 6. 인증 및 세션 분리 기준

- 가족 진입 성공: 가족 홈 열람 컨텍스트 획득
- 계정 로그인 성공: 관리자/개인화 기능 세션 획득

권장 해석:

- 가족 공유 비밀번호만 통과한 사용자는 읽기 또는 제한된 작성만 가능하게 시작한다.
- 공지 관리, 멤버 관리, 설정 변경은 계정 로그인 세션이 있어야 한다.

## 7. 아직 확정 전인 항목

- `guest`에게 투두 읽기를 기본 허용할지
- `child`에게 일반 글 작성까지 열지
- `member`에게 가족 공용 일정 삭제를 허용할지
- 댓글 기능을 MVP에 넣을지
- 일기 공개 범위를 개인/가족/선택 공개로 나눌지

## 8. `02`와 `06`의 사용 방식

### `02 Entry/Auth`

- 화면 가드
- 로그인 필요 메시지
- 접근 실패 분기
- 역할별 CTA 노출

### `06 Platform/Admin`

- 권한 설정 UI
- 역할 설명 문구
- 모듈 접근 허용 범위 설정
- 관리자 기능 메뉴 가드

## 9. 변경 규칙

- 역할 추가 또는 삭제는 이 문서와 `REPORT.md`를 같이 갱신한다.
- `메인 노출 설정`, `고정 설정`, `모듈 on/off` 권한은 별도 검토 없이 넓히지 않는다.
