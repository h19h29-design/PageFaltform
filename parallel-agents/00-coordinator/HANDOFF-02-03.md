# 02-03 Handoff

목적:

- `01 Foundation`이 별도 진행 중인 상황에서 `02 Entry/Auth`와 `03 Dashboard`가 겹치지 않게 바로 착수할 수 있도록 기준선을 제공한다.
- 이 문서는 구현 문서가 아니라 조율 문서다.

## 1. `01 Foundation`에서 받아야 하는 최소 결과물

아래 5가지만 오면 `02`, `03`은 큰 재작업 없이 시작할 수 있다.

1. 워크스페이스 구조와 패키지 이름
2. `apps/web`의 기본 라우트 골격
3. DB v1의 핵심 엔티티 초안
4. 공용 env 이름 규칙
5. 공용 TS 설정과 import 경로 규칙

권장 전달 포맷:

- 사용 패키지 매니저
- 실행 명령
- 생성된 앱/패키지 목록
- Prisma 스키마에 들어간 핵심 모델
- 아직 미정인 항목

## 2. `02 Entry/Auth`가 먼저 고정할 것

### 라우트 책임

- `/`: 플랫폼 랜딩 또는 가족 진입 유도
- `/f/[familySlug]`: 가족 스플래시 + 접근 코드/비밀번호 진입
- `/app/[familySlug]`: 가족 홈과 모듈 진입의 기본 셸
- `/console`: 관리자/운영자 전용 영역

### 인증 분리 원칙

- 가족 진입용 접근 수단과 관리자 계정 인증은 분리한다.
- 가족 공유 비밀번호는 편의 접근이다.
- 관리자 기능 보호는 계정 기반 인증으로 간다.

### `02`가 먼저 문서화해야 하는 것

1. 비로그인 허용 범위
2. 역할별 접근 가능 행동
3. 진입 실패 상태와 예외 흐름
4. 모바일 우선 화면 전환 순서

## 3. `03 Dashboard`가 먼저 고정할 것

### 홈 섹션 기본안

- 히어로
- 오늘 일정
- 오늘 할 일
- 중요 공지
- 진행률/습관
- 최근 글/사진
- 고정 카드

### 카드 규칙 기본안

- 모든 카드에는 `type`, `priority`, `featured`, `pinned`, `displayStartsAt`, `displayEndsAt` 개념이 필요하다.
- `03`은 실제 UI보다 먼저 카드 슬롯과 정렬 규칙을 문서화한다.
- `04`, `05`, `07`은 각자 카드 UI를 만들기 전에 요약 payload를 먼저 맞춘다.

### `03`이 먼저 문서화해야 하는 것

1. 메인 홈 섹션 순서
2. 같은 우선순위일 때 정렬 기준
3. 오늘 기준 집계의 시간 범위
4. 모바일에서 카드 최대 개수

## 4. `02`와 `03`의 공유 계약 초안

### 가족 컨텍스트

- 현재 family slug
- tenant id
- 활성 모듈 목록
- 현재 사용자 역할
- 로그인 상태

### 대시보드 카드 최소 공통 필드

- `id`
- `tenantId`
- `moduleKey`
- `cardType`
- `title`
- `summary`
- `priority`
- `featured`
- `pinned`
- `displayStartsAt`
- `displayEndsAt`
- `visibilityScope`
- `href`

### 권한 판단 최소 축

- 읽기 가능
- 작성 가능
- 수정 가능
- 관리 가능
- 메인 노출 설정 가능

## 5. `02`가 지금 바로 할 수 있는 일

- 사용자 여정 문서화
- 화면 목록과 상태표 정리
- 권한 매트릭스 초안 작성
- 접근 코드/비밀번호와 관리자 로그인 분리 설계

## 6. `03`이 지금 바로 할 수 있는 일

- 메인 홈 정보 구조 설계
- 카드 슬롯 체계 정의
- 모듈별 요약 payload 인터페이스 초안 작성
- 홈 우선순위 기본 정책 정리

## 7. `02`와 `03`이 지금 하면 안 되는 일

- `01`의 실제 폴더/패키지 구조가 나오기 전에 import 경로를 확정하는 일
- DB 스키마를 각자 따로 늘리는 일
- 공용 UI 토큰 없이 개별 스타일을 먼저 확정하는 일
- 권한 규칙 없이 관리 기능 화면부터 그리는 일

## 8. 임시 기본값

- 첫 화면 톤: 따뜻하고 감성적인 가족 사진 중심
- 홈 우선 카드: 중요 공지 -> 오늘 일정 -> 오늘 할 일
- 권한 기본안: `owner > admin > member > guest`, `child`는 제한형
- 공지 기본안: 중요 전달용, 댓글 기본 비활성, 읽음 확인은 중요/긴급 위주

## 9. 사용자 확인이 들어오기 전까지의 합의안

- `02`는 관리자 보안과 가족 진입 UX를 반드시 분리한다.
- `03`은 공지/일정/투두/진행률이 모두 올라올 수 있는 구조로 먼저 설계한다.
- `02`, `03` 모두 구현보다 문서 계약을 먼저 맞춘다.
