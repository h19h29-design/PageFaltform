현재 상태 요약:

- 프로젝트는 아직 구현 전 단계이며, 문서 기준 구조와 우선순위가 먼저 잡혀 있다.
- 초기 구현 전략은 `모듈형 모놀리식 + monorepo`다.
- 병렬 작업은 가능하지만, 공통 계약과 권한/테넌트 규칙을 먼저 고정하지 않으면 충돌 확률이 높다.

## 1. 역할별 의존성 요약

| 스트림 | 핵심 책임 | 하드 의존성 | 이 스트림이 먼저 고정해야 하는 계약 | 바로 뒤따르는 스트림 |
| --- | --- | --- | --- | --- |
| `01 Foundation` | monorepo, 앱 골격, 공용 설정, DB 기초 스키마 | 없음 | 워크스페이스 구조, 패키지 연결, DB v1, env 규칙, 공통 타입 베이스 | `02`, `03`, `06`, 나머지 전체 |
| `02 Entry and Auth` | 스플래시, 가족 진입, 접근 코드, 관리자 인증, 권한 가드 | `01` | 라우트 골격, 세션 방식, 가족 접근과 관리자 인증 분리, 역할 모델 초안 | `03`, `06` |
| `03 Dashboard Home` | 가족 홈 레이아웃, 위젯 시스템, 메인 카드 규칙 | `01`, `02` | 홈 섹션 구조, 카드 슬롯, 모듈->대시보드 집계 인터페이스, 우선순위 규칙 | `04`, `05`, `07`, `06` |
| `04 Content Modules` | 공지, 일반 글, 갤러리, 일기 | `01`, `03` | 콘텐츠형 폼 패턴, 공지 전용 필드, 메인 노출 매핑 | `03`, `06`과 상호 연동 |
| `05 Schedule Modules` | 투두, 일정, 시간표, 하루 계획 | `01`, `03` | 오늘 기준 집계 규칙, 시간 데이터 규칙, 개인/가족 스코프 구분 | `03`, `06`과 상호 연동 |
| `07 Tracker and Routines` | 진행률, 습관, 연속 기록, 목표 카드 | `01`, `03` | 퍼센트/개수/연속형 데이터 규칙, 카드 요약 스키마, 집계 방식 | `03`, `06`과 상호 연동 |
| `06 Platform and Deploy` | 가족 설정, 테마, 모듈 on/off, 운영/배포 | `01`, `02`, `03` | 테넌트 설정 스키마, 권한 정책, 활성 모듈 구조, 운영 환경 기준 | 전 스트림 마감 단계와 연결 |

추가 메모:

- `04`, `05`, `07`은 문서상으로는 병행 가능하지만, 실제로는 `03 Dashboard Home`의 집계 계약이 먼저 안정화되어야 안전하다.
- `06 Platform and Deploy`는 마지막 순서가 맞지만, 배포용 env 분리와 기본 CI 틀은 `01 Foundation`과 함께 초기에 아주 얇게 시작해도 된다.
- `packages/modules/core`, `packages/dashboard`, `packages/auth`, `packages/tenant`, `packages/database`, `packages/ui`는 사실상 공용 계약 지대라서 단독 소유처럼 다루면 안 된다.

## 2. 추천 착수 순서

1. `01 Foundation` 단독 선행
2. `02 Entry and Auth` 착수
3. `03 Dashboard Home` 착수
4. `04 Content Modules`와 `05 Schedule Modules` 병행
5. `07 Tracker and Routines` 병행
6. `06 Platform and Deploy` 본격 착수

병행 기준:

- `01` 완료 전에는 다른 스트림이 구조를 확정하지 않는다.
- `02`는 `01`의 라우트/패키지/DB 기초가 생기면 바로 시작할 수 있다.
- `03`은 `02`가 가족 홈 진입 경로와 테넌트 컨텍스트를 최소 수준으로 열어준 뒤 시작하는 것이 안전하다.
- `04`, `05`, `07`은 모두 `03`의 위젯 계약 v1이 나온 뒤 병행한다.
- `06`은 구현은 뒤로 두되, 설정 항목 목록과 인프라 체크리스트 문서화는 앞에서부터 병행 가능하다.

## 3. 충돌 가능성이 큰 지점

### A. 공통 메타데이터 계약

영향 범위:

- `packages/modules/core`
- `packages/dashboard`
- `packages/database`
- `packages/modules/*`

충돌 이유:

- `featuredOnHome`, `pinnedOnHome`, `homePriority`, `displayStartsAt`, `displayEndsAt`, `visibilityScope`, `attachments`, `commentsEnabled`, `readTrackingEnabled`는 거의 모든 모듈과 대시보드가 공유한다.

방지 메모:

- 이 필드 집합을 바꾸는 작업은 반드시 `docs/architecture.md`를 먼저 갱신한다.
- DB 스키마 변경은 `01 Foundation`이 최종 반영하되, 요구사항 제안은 각 스트림이 문서로 남긴다.

### B. 권한 모델과 가족 진입 방식

영향 범위:

- `packages/auth`
- `packages/tenant`
- `packages/platform`
- `apps/web`

충돌 이유:

- 가족 접근 코드/비밀번호와 관리자 계정 인증이 서로 다른 흐름인데, 어느 화면까지 비로그인 허용인지 아직 확정되지 않았다.
- `owner / admin / member / guest / child` 권한 차이가 세부 행동 단위까지 내려오지 않았다.

방지 메모:

- 관리자 보호는 계정 인증 기준, 가족 진입은 공유 접근 장치 기준으로 분리 고정한다.
- CRUD, 읽기, 댓글, 읽음 확인, 모듈 설정 권한을 역할별 매트릭스로 먼저 정리한다.

### C. 대시보드 카드 규칙

영향 범위:

- `packages/dashboard`
- `packages/modules/core`
- `packages/modules/announcements`
- `packages/modules/todo`
- `packages/modules/calendar`
- `packages/modules/progress`
- `packages/modules/habits`

충돌 이유:

- "메인에 무엇이 먼저 보여야 하는가"와 "오늘 기준 집계를 어떻게 계산하는가"가 아직 미확정이다.
- 같은 카드 슬롯을 공지, 일정, 투두, 진행률이 동시에 요구할 가능성이 높다.

방지 메모:

- `03 Dashboard Home`이 카드 슬롯과 섹션 순서를 먼저 고정한다.
- 각 모듈은 화면 구현보다 먼저 "대시보드 요약 payload"를 맞춘다.

### D. 공지 게시판의 실제 사용 방식

영향 범위:

- `packages/modules/announcements`
- `packages/dashboard`
- `packages/auth`

충돌 이유:

- 공지가 단순 소식 피드인지, 읽음 확인이 필요한 전달 채널인지에 따라 필드와 UI가 크게 달라진다.
- 댓글 허용 여부, 읽음 확인 의무 여부, 대상 범위 설정 방식이 미확정이다.

방지 메모:

- 공지는 MVP에서 "전달/확인 중심"으로 두고, 일반 소식 피드는 일반 글 게시판으로 분리하는 기본안을 유지한다.
- 읽음 확인은 전체 공지 기본값이 아니라 중요/긴급 공지 우선 적용으로 시작하는 편이 안전하다.

### E. UI 토큰과 첫 화면 감성 톤

영향 범위:

- `packages/ui`
- `apps/web`
- `02`, `03` 전반

충돌 이유:

- 감성 톤과 홈 분위기가 확정되지 않으면, 스플래시와 홈이 서로 다른 제품처럼 보일 수 있다.

방지 메모:

- `02`와 `03`이 따로 스타일을 만들지 말고, `01`이 만든 토큰 기반에 임시 테마를 하나 합의해서 사용한다.

### F. 모듈 on/off와 메뉴 구조

영향 범위:

- `packages/platform`
- `packages/tenant`
- `packages/dashboard`
- `apps/web`

충돌 이유:

- 모듈 비활성화 시 홈 카드, 메뉴, 라우트 보호가 같이 바뀌어야 한다.

방지 메모:

- `EnabledModule`과 메뉴 구성은 설정 데이터로 관리하고, 각 스트림이 하드코딩을 늘리지 않는다.

## 4. 사용자 확인 질문

### 우선 확인해야 할 질문

1. 첫 화면은 사진 중심의 따뜻한 감성형에 가까운지, 깔끔한 정보형에 가까운지 확인이 필요하다.
2. 가족 홈 첫 화면에서 가장 먼저 보여야 하는 카드 3개가 무엇인지 확인이 필요하다.
3. `owner`, `admin`, `member`, `guest`, `child`가 각각 어디까지 수정/관리할 수 있는지 확인이 필요하다.
4. 공지 게시판이 "중요 전달 + 읽음 확인" 중심인지, "일반 소식 피드" 중심인지 확인이 필요하다.

### 작업을 덜 막히게 하는 추가 질문

1. 가족 비밀번호는 가족 전체가 공유하는 단일 비밀번호인지, 구성원별 접근 코드 발급도 필요한지 확인이 필요하다.
2. 일정과 투두는 가족 공용 비중이 큰지, 개인별 비중이 큰지 확인이 필요하다.
3. 학교/학원 시간표와 하루 계획표는 개인 전용인지, 가족 홈에도 합산 노출할지 확인이 필요하다.
4. 진행률/습관은 개인 루틴과 가족 루틴을 분리해서 시작할지 확인이 필요하다.
5. 초기 배포 우선순위가 Vercel인지, 다른 환경도 동시에 고려할지 확인이 필요하다.

## 5. 임시 기본값

사용자 확인 전까지는 아래 가정으로 진행하면 재작업을 줄일 수 있다.

- 첫 화면 톤: 따뜻하고 감성적인 사진 중심, 모바일 우선, 크림/베이지 계열
- 가족 홈 카드 우선순위: 중요 공지 -> 오늘 일정 -> 오늘 할 일 -> 진행률/습관 -> 최근 사진/글
- 권한 기본안:
  - `owner`: 가족 설정, 멤버/권한 관리, 모듈 on/off 가능
  - `admin`: 게시판/일정/공지 관리 가능, 가족 전체 설정은 제한
  - `member`: 일반 작성/수정 가능, 설정 불가
  - `guest`: 읽기 중심
  - `child`: 허용된 일부 모듈만 접근
- 공지 기본안: 중요 전달 채널로 시작, 읽음 확인은 중요/긴급 공지만 우선 적용, 댓글은 기본 비활성

## 6. 스트림별 시작 체크리스트

### `01 Foundation`

- 워크스페이스 매니저와 기본 명령 체계를 확정한다.
- `apps/web`, `packages/*`, `tests`, `infra`가 연결되는 monorepo 골격을 만든다.
- Prisma 기반 DB 스키마 v1에 `PlatformUser`, `FamilyTenant`, `Membership`, `EnabledModule` 뼈대를 잡는다.
- 공용 TS 설정, lint, env 샘플, seed/sample data 기준을 만든다.

### `02 Entry and Auth`

- `/`, `/f/[familySlug]`, `/app/[familySlug]`, `/console` 라우트 책임을 문서로 고정한다.
- 가족 진입 플로우와 관리자 로그인 플로우를 분리한 화면 상태표를 만든다.
- 역할별 접근 가능 행동 표를 초안으로 만든다.
- `packages/auth`와 `packages/tenant`가 공유할 세션/컨텍스트 API를 정의한다.

### `03 Dashboard Home`

- 홈 섹션을 `히어로`, `오늘`, `진행`, `최신`, `특별`, `고정`으로 둘지 여부를 먼저 고정한다.
- 카드 슬롯과 정렬 규칙, 우선순위 숫자 체계를 정의한다.
- 모듈이 보내야 할 "대시보드 요약 payload" 인터페이스를 만든다.
- 모바일 우선 레이아웃과 공통 카드 UI 베이스를 정한다.

### `04 Content Modules`

- 공지와 일반 글의 차이를 필드와 행동 기준으로 분리한다.
- 공지 필드 세트와 리스트 노출 항목을 먼저 고정한다.
- 공통 작성 폼에서 `메인에 보이기`, `고정`, `노출 기간`을 재사용 가능하게 설계한다.
- 갤러리/일기의 공개 범위와 댓글 정책을 임시 기본값으로 맞춘다.

### `05 Schedule Modules`

- `오늘` 기준 집계 규칙과 타임존 기준을 명시한다.
- 일정, 시간표, 하루 계획표의 데이터 경계를 나눈다.
- 투두를 단순 체크형으로 시작할지, 담당자 기반으로 갈지 임시 결정을 둔다.
- 홈 카드에 올릴 최소 요약 정보 세트를 고정한다.

### `07 Tracker and Routines`

- 퍼센트형, 개수형, 연속형의 계산식과 저장 필드를 먼저 정의한다.
- 홈 카드 요약값이 어떤 주기로 계산되는지 정한다.
- 개인 루틴과 가족 루틴을 분리 가능한 구조로 시작한다.
- 습관 체크와 진행률 보드가 공통 메타데이터 계약을 따르도록 맞춘다.

### `06 Platform and Deploy`

- 가족 설정 화면에서 바꿀 수 있는 항목 목록을 먼저 문서화한다.
- 테마, 메뉴, 모듈 on/off를 데이터 기반 설정으로 설계한다.
- 환경변수 표, 배포 타깃, 스토리지/이미지 전략을 최소 수준으로 정리한다.
- 테스트 범위를 진입, 홈, 공지, 모듈 on/off 기준으로 우선 정의한다.

## 7. 충돌 방지 운영 메모

- 공통 계약 변경은 구현보다 문서가 먼저다.
- `packages/modules/core`, `packages/dashboard`, `packages/auth`, `packages/tenant`, `packages/database`, `packages/ui` 변경은 스트림 단독 판단으로 확정하지 않는다.
- 카드 노출 규칙과 권한 규칙은 샘플 UI보다 먼저 맞춘다.
- `04`, `05`, `07`이 DB 스키마를 직접 따로 늘리기보다 요구 필드를 `01`에 제안하는 흐름으로 맞춘다.
- 공지, 투두, 일정, 진행률은 전부 홈 카드 집계와 연결되므로 API보다 "요약 인터페이스"를 먼저 맞춘다.
