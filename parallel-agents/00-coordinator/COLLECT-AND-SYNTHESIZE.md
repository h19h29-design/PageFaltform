# Collect and Synthesize Playbook

목적:

- 다른 스트림이 "완료" 또는 "초안 완료" 상태가 되었을 때 coordinator가 어떤 순서로 결과를 회수하고, 무엇을 확인하고, 어떤 형식으로 최종 산출물을 만들지 고정한다.
- 이 문서는 실행용 운영 절차다.

## 1. 기본 운영 원칙

1. 스트림별 완료 선언만 받지 말고, 결과물 묶음을 함께 받는다.
2. 공통 계약에 닿는 변경은 단순 완료로 처리하지 않고 충돌 검토를 먼저 한다.
3. 사용자에게 보여줄 때는 "변경 파일 목록"보다 "보여줄 수 있는 결과"를 먼저 정리한다.
4. 샘플은 글만이 아니라 화면 흐름, 카드 데이터, 권한 차이처럼 검토 가능한 단위로 만든다.

## 2. 회수 순서

1. `01 Foundation`
2. `02 Entry/Auth`
3. `03 Dashboard`
4. `04 Content Modules`
5. `05 Schedule Modules`
6. `07 Tracker and Routines`
7. `06 Platform/Admin/Deploy`

메모:

- `04`, `05`, `07`은 병렬 회수 가능하지만, 합칠 때는 `03 Dashboard` 계약을 기준으로 재정렬한다.
- `06`은 기능 마감보다 설정/운영 관점 확인용으로 마지막에 묶는다.

## 3. 각 스트림에서 꼭 받아야 하는 것

각 스트림은 완료 시 아래 6가지를 제출해야 한다.

1. 이번 작업 범위 요약 5줄 이내
2. 수정한 파일 경로 목록
3. 바뀐 공통 계약 또는 새 계약
4. 남은 리스크와 미확정 항목
5. 바로 보여줄 수 있는 샘플 1개 이상
6. 다음 스트림이 이어받아야 할 핸드오프 메모

## 4. 완료 수거 체크리스트

### `01 Foundation`

- 워크스페이스 구조가 문서와 일치하는지
- 앱 실행 명령이 통하는지
- DB v1 핵심 모델이 존재하는지
- `02`, `03`이 붙을 import/env 기준이 정리되었는지

### `02 Entry/Auth`

- 가족 진입 흐름과 관리자 로그인 흐름이 분리되었는지
- 역할 가드 기준이 문서와 맞는지
- 진입 실패/권한 부족 상태가 정의되었는지
- 사용자에게 보여줄 화면 흐름 샘플이 있는지

### `03 Dashboard`

- 홈 섹션 구조가 계약과 맞는지
- 카드 payload가 공통 필드를 따르는지
- 모바일/데스크톱 노출량이 정리되었는지
- 샘플 카드 데이터나 화면 예시가 있는지

### `04 Content Modules`

- 공지와 일반 글이 구조적으로 분리되었는지
- 공지 필드가 전달/확인 목적에 맞는지
- 홈 노출용 payload를 만들 수 있는지
- 샘플 게시물/공지 시나리오가 있는지

### `05 Schedule Modules`

- 일정/투두/시간표 경계가 분명한지
- 오늘 기준 집계가 가능한지
- 홈 카드로 올릴 최소 요약 정보가 있는지
- 샘플 오늘 화면 데이터가 있는지

### `07 Tracker and Routines`

- 퍼센트/개수/연속형 규칙이 분명한지
- 홈 카드 요약이 가능한지
- 체크와 정의 권한이 구분되는지
- 샘플 진행 카드가 있는지

### `06 Platform/Admin/Deploy`

- 가족 설정 가능 항목이 표로 정리되었는지
- 모듈 on/off와 권한 설정이 데이터 기반인지
- 배포 체크리스트와 env 표가 있는지
- 운영자 관점 샘플 설정 화면 또는 설정 데이터가 있는지

## 5. 충돌 검토 포인트

아래 항목에 닿으면 통합 전에 반드시 재검토한다.

- DB 스키마 공통 필드 변경
- 역할 추가/삭제
- 카드 공통 필드 변경
- 섹션 이름 변경
- 테넌트 식별 방식 변경
- 모듈 on/off 동작 기준 변경

## 6. 통합 산출물 패키지 구조

다른 스트림 결과를 받은 뒤 coordinator는 최종적으로 아래 묶음을 만든다.

1. 완료 현황 요약
2. 공통 계약 확정본
3. 충돌 해결 메모
4. 사용자 검토가 필요한 남은 질문
5. 바로 볼 수 있는 샘플
6. 다음 구현 우선순위

## 7. 사용자에게 보여줄 샘플 유형

최소 3개는 준비한다.

1. 진입 흐름 샘플
2. 가족 홈 카드 샘플
3. 권한 차이 샘플

가능하면 추가:

4. 공지 작성/읽음 확인 샘플
5. 오늘 일정/투두 집계 샘플
6. 진행률/습관 카드 샘플

## 8. 통합 후 최종 체크

1. `owner/admin/member/guest/child` 해석이 문서마다 다르지 않은지
2. 홈 카드 필드 이름이 모듈마다 다르지 않은지
3. 홈 우선순위가 공지/일정/투두/진행률 기준과 충돌하지 않는지
4. 가족 진입과 관리자 인증이 다시 섞이지 않았는지
5. 샘플이 실제 구조와 어긋난 기획용 문장만 남지 않았는지

## 9. coordinator 최종 응답 형식

사용자에게는 아래 순서로 보여준다.

1. 무엇이 끝났는지
2. 지금 바로 볼 수 있는 샘플
3. 아직 남은 리스크
4. 다음으로 붙일 스트림

## 10. 빠른 회수용 메시지 초안

각 스트림이 끝났을 때 coordinator가 요청할 수 있는 기본 문구:

> 작업 완료본을 넘길 때는 요약, 수정 파일, 공통 계약 변경, 남은 리스크, 샘플 1개, 다음 핸드오프 메모까지 같이 주세요.

## 11. 통합 실패 조건

아래 중 하나면 "완료"가 아니라 "보완 필요"로 본다.

- 샘플이 전혀 없음
- 공통 계약 변경을 문서화하지 않음
- 다음 스트림이 이어받을 기준이 없음
- 권한 또는 카드 구조가 기존 coordinator 문서와 충돌함
